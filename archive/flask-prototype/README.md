# Flask プロトタイプ（参考用・凍結）

これは「Slay the Spinner」のゲームデザインを検証した最初の実装です。
物理演算はサーバーサイドで一括計算し、クライアントはCSS `@keyframes`で
その結果を再生するだけの仕組みで、リアルタイムなクライアントゲームでは
ありません。ブラウザ／Steam配信を目指す本実装は `godot/` 以下で
Godotベースに作り直されています。

**このディレクトリは参考資料としてのみ残しています。機能追加はしないでください。**
新しい実装で移植・再検証する際の仕様リファレンス、および挙動の
突き合わせ用（`simulation.py`をそのまま実行して基準トラジェクトリを
生成し、Godot版の物理演算と数値比較する等）に使います。

## ローカルでの起動方法

```bash
python3 -m venv .venv
.venv/bin/pip install flask numpy
.venv/bin/python app.py
# http://127.0.0.1:5000/ で確認
```

## 移植の際に見つかった食い違い

- **`ENEMY_LIST` の位置と速度は死んだデータ**。`app.py` は `initial_conditions`
  の固定値（全敵が中央(5,5)から(3,4)）を使っており、`Enemy.pos_x/pos_y/vel_x/vel_y`
  は一度も読まれない。ただし値はレベルが上がるほど大きく、強い敵ほど速く
  発射させる意図は読み取れる。
- **`run_simulation` の `decay` 引数も使われていない**。`app.py` は0.99を渡すが
  本文は `object.decay` を使う。昔は `vel *= decay`（大きいほど良い）だった名残と
  思われ、そのせいで `custom_part.py` の "Full Steam Ahead"（速度減衰を改善）が
  実際には遅くなる効果になっている。
- **パーツ説明が実際の値と食い違っている**。"Spin Engine" は「10%上昇」と
  書いてあるが `rps_value=1.2`＝20%上昇。
- `maptree.py` の検証で、ノード82へ到達できるかの判定に `node_73` の
  `"right"` を見ているが、73から82へは `left` で届く（`right` は範囲外の84）。
  73の寄与が無視され、正しいマップが誤って棄却されることがある。

（以前ここに「壁のSVGポリゴンが4枚重なって中央を塗り潰すバグがある」と
書いていたが、**誤りだったので取り消す**。頂点を計算すると各壁は辺に沿った
深さ1単位の平たい三角形で、中央には届かない。プレイ中に中央を覆って見えた
マゼンタの塊は、衝突エフェクト（1pxの円を`scale(1000)`＝直径1000pxまで広げる）
の方だった。）
